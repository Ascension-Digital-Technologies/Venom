# Build Venom from Source

> **Applies to:** Venom 1.1.0  
> **Purpose:** produce the production compiler from the released source package

Normal users may use a prebuilt release when one is available. Building from source is supported for teams that require reproducible local binaries, internal qualification, or platform-specific packaging.

## Requirements

### Windows

- Windows 10 or Windows 11
- Visual Studio with **Desktop development with C++**
- CMake 3.20 or newer
- Python 3.10 or newer

### Linux

- A C++17 compiler such as Clang or GCC
- CMake 3.20 or newer
- Ninja or Make
- Python 3.10 or newer

The normal compiler build uses the verified TurboJS/WASM runtime embedded in the source package. Emscripten is not required unless intentionally rebuilding that embedded runtime artifact.

## Windows build

From a PowerShell prompt in the extracted repository:

```powershell
.\scripts\windows\build.ps1 -Config Release
```

The default Visual Studio output is usually:

```text
build\Release\venom.exe
```

Verify the binary:

```powershell
.\build\Release\venom.exe --version
.\build\Release\venom.exe doctor --profile production
```

For an explicit clean build:

```powershell
.\scripts\windows\clean.ps1 -NoPause
.\scripts\windows\build.ps1 -Config Release
```

## Linux build

```bash
bash scripts/linux/build.sh --config Release
```

Verify the binary:

```bash
./build/venom --version
./build/venom doctor --profile production
```

Depending on the selected generator, the executable may be placed in a configuration-specific subdirectory. The build script reports the resolved path.

## Direct CMake workflow

The scripts are the supported convenience path, but direct CMake use is also available:

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel
```

For a multi-configuration generator such as Visual Studio:

```powershell
cmake -S . -B build
cmake --build build --config Release --parallel
```

## JavaScript hardener

Production hardening is built directly into the native `venom` executable. The compiler runs pinned Terser 5.49.0 and `javascript-obfuscator` 5.4.7 bundles inside its linked TurboJS engine, so production builds do not require Node.js, npm, `node_modules`, temporary JavaScript files, or an external hardener process.


## Build acceleration

Venom supports compiler caching, precompiled headers, parallel MSVC compilation, and separately controlled interprocedural optimization. The default scripts choose safe production settings. Teams measuring build performance should use the reproducible workflow in [Build performance](../operations/build-performance.md).

## Validation after building

Run the production doctor first:

```powershell
venom doctor --profile production
```

Then run the full native test suite:

```powershell
ctest --test-dir build -C Release --output-on-failure -j 1
```

Use `-C Release` for multi-configuration generators. A sequential run is recommended for final release evidence because several tests create production distributions and intentionally perform heavier qualification work.

## Build failures

Common causes include:

- the JS hardener was not installed;
- a compiler without C++17 support was selected;
- CMake or Python is below the supported version;
- a stale build directory was generated by a different compiler;
- endpoint protection quarantined a generated test executable;
- Windows path length or quoting altered a toolchain invocation.

For diagnosis, remove the build directory, run `doctor --profile production`, and rebuild using the platform script. See [Debugging](../guides/debugging.md) for runtime and distribution-specific failures.

## Rebuilding the embedded TurboJS/WASM runtime

Normal users do not need Emscripten because the verified TurboJS/WASM runtime is embedded in the source package. Runtime contributors can rebuild it with the canonical controller:

### Windows

```powershell
.\scripts\windows\build-emsdk.bat -OutDir build\emscripten-build -Clean -Force
```

### Linux and macOS

```bash
bash scripts/linux/build-emsdk.sh --out-dir build/emscripten-build --clean --force
```

The controller performs Emscripten setup, a strict TurboJS/WASM preflight, runtime compilation, ABI verification, embedding, native compiler rebuild, and protected-runtime smoke verification. The output directory is a controller workspace; the actual runtime artifact is written once under `turbojs-wasm/` without recursively nesting that directory.

### Cutover rule

A rebuilt runtime is not release-ready until the `verify-runtime --require-real-engine` gate passes for the production distribution:

```bash
venom verify-runtime <dist> --require-real-engine
```

The checked-in runtime snapshot must be regenerated only through the canonical controller and must remain synchronized with its generated C/C++ embedding.


### Windows command invocation safety

The controller invokes `emcc.exe` and PowerShell scripts through argument arrays rather than constructing a `cmd.exe` command string. This prevents Windows from treating quote characters as part of the executable name and avoids failures when the repository path contains spaces.

Use preflight-only mode to validate the pinned toolchain and repository inputs without rebuilding the runtime:

```powershell
.\scripts\windows\build-emsdk.bat -OutDir build\emscripten-build -PreflightOnly -AllowMissing
```


> **Verified WASM toolchain:** Venom pins Emscripten 4.0.10 for reproducible release builds. The minimal TurboJS/WASM target excludes TurboJS POSIX libc helpers, allowing newer Emscripten releases to compile the web/worker runtime without `environ` or `sighandler_t` compatibility failures.
